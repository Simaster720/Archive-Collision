# 트래킹 박스 지터는 클라이언트 측에서 매 조회마다 재생성하고, 렌더 좌표를 저장하지 않는다

> 상태: Accepted (2026-06-15) · 관련: [CONTEXT.md](../../CONTEXT.md) (트래킹 박스·AI 판별) · [PLAN-ai-verdict-0615](../PLAN-ai-verdict-0615.md)

## 맥락

작업의 개념적 핵심은 진짜 자료를 관객이 "AI가 지금 분석 중"이라고 의심하게 만드는 데 있다. 데이터(`eval_results.xlsx`의 `특징JSON`)는 박스마다 고정된 픽셀 좌표(`x1,y1,x2,y2`)를 갖는다. 이 좌표를 매번 동일하게 렌더하면 관객이 **"저장된 데이터를 불러오는 것"** 으로 인지해 실시간 분석이라는 환상이 깨진다(클라이언트 명시 요구).

해결책은 박스를 매 조회마다 **미세하게 다르게** 그리는 것이다. 같은 자료를 다시 봐도 박스 모양이 조금씩 달라져, 좌표가 고정 저장된 게 아니라 매번 계산되는 듯 보이게 한다.

## 결정

- **지터는 클라이언트 측 렌더 시점에 적용한다.** 데이터의 원본 좌표는 그대로 두고, 표시 직전에 좌·우 모서리를 각각 독립적으로 ±랜덤 변형한다(높이 고정, 중심은 대략 유지되되 약간의 드리프트 허용).
- **재생성 주기 = 상세 진입(컴포넌트 마운트)마다 1회.** 재방문/재내비게이션 시 다시 랜덤화한다(매번 새로 분석되는 인상). 세션 저장으로 억제하지 않는다.
- **변형된 좌표는 어디에도 저장하지 않는다.** 빌드 산출물(`collection.json`/`file.ai`)에는 원본 좌표만 담고, 지터는 순수 표시 효과로만 존재한다.
- 거의 전폭(near-full-width)인 박스는 지터 폭을 감쇄해 깨짐을 막는다. 지터 폭·빈도는 시각 튜닝 대상(코드 상수)이다.

## 고려한 대안

- **빌드타임에 지터를 구워 좌표에 저장** — 재현 가능하고 SSG 친화적이나, 자료당 박스가 영구 고정되어 "저장된 데이터" 인상을 그대로 남긴다(요구의 정반대). 기각.
- **연속 떨림(tremor) 애니메이션** (handoff 데모의 `enableJitter`) — 매 프레임 흔들려 산만하고, 클라이언트가 원한 건 "조회마다 다른 모양"이지 "계속 떨림"이 아니다. 기각.
- **중심 완전 고정 + 대칭 너비만 변형** — 더 안전하지만 "약간 어긋나도 OK"라는 허용 범위를 못 살려 덜 유기적. 독립 모서리 방식 채택.

## 결과

- 박스 위치/크기로 **정밀한 객체 위치 주장을 하지 않는다**(이미 박스 ~25/35개가 전체화면 영역이라 정밀도는 의미 없음). 지터는 이 비정밀성을 오히려 연출로 전환한다.
- 후대 개발자 주의: 박스가 매 조회마다 달라지는 것은 **버그가 아니라 의도**다. 좌표를 "안정화/고정"하지 말 것 — 그 순간 작업의 개념이 무너진다. (이 ADR이 그 이유다.)
- 지터는 표시 전용이라 데이터 파이프라인(`build-data.mjs`)과 분리된다. 박진혁 시트로 판별 소스를 교체해도 지터 로직은 영향받지 않는다.
